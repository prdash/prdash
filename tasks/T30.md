## Objective

Add a new "Ready to PR" query group that detects the user's recently-pushed branches (last 7 days) without an associated open PR, and lets the user open GitHub's new-PR page directly from the dashboard.

## Background

GitHub shows a "create PR" banner when you push a branch that doesn't have an open PR yet. This feature brings that same detection into the terminal dashboard. It surfaces candidate branches as a collapsible group alongside existing PR groups. Pressing Enter on a branch opens the GitHub compare page (`/compare/{branch}?expand=1`) pre-filled for PR creation, keeping the dashboard read-only.

## Requirements

### Data Layer

1. Add `CandidateBranch` frozen Pydantic model to `models.py` with fields: `name` (str), `last_commit_date` (datetime), `compare_url` (str), and `age_display` computed field (same logic as `PullRequest.age_display`)
2. Add `branches: list[CandidateBranch] = Field(default_factory=list)` field to `QueryGroupResult`
3. Add `parse_refs_results(data, username, owner, repo, default_branch)` parser to `models.py` that:
   - Filters refs where `associatedPullRequests.totalCount == 0`
   - Filters by `target.author.user.login == username`
   - Filters by `committedDate` within 7 days
   - Excludes the repository's default branch
   - URL-encodes branch names with slashes (`urllib.parse.quote`)
4. Update `deduplicate_groups` to pass `branches` through unchanged (branches can't overlap with PRs)

### Config

5. Add `READY_TO_PR = "ready_to_pr"` to `QueryGroupType` enum in `config.py`
6. Add default entry to `DEFAULT_QUERY_GROUPS` (last position, enabled, name: "Ready to PR")

### GitHub API

7. Add `BRANCH_REFS_QUERY` GraphQL query to `queries.py` that fetches:
   - `repository.defaultBranchRef.name` (to exclude main/master dynamically)
   - `repository.refs(refPrefix: "refs/heads/", first: 100, orderBy: {field: TAG_COMMIT_DATE, direction: DESC})` with each ref's `name`, `target { ... on Commit { committedDate, author { user { login } } } }`, and `associatedPullRequests(states: OPEN, first: 1) { totalCount }`
8. Add `fetch_candidate_branches(config, group)` async method to `GitHubClient` that executes the query and calls `parse_refs_results`
9. Update `fetch_all_groups` to dispatch to `fetch_candidate_branches` when `group.type == READY_TO_PR`

### Widget Layer

10. Add `BranchSelected(Message)` with `branch: CandidateBranch` field to `pr_list.py`
11. Add `BranchRow(ListItem)` widget showing branch name + age + "[cyan]ready to PR[/cyan]" meta line
12. Update `_rebuild_list` in `PRListWidget`:
    - Group header count includes both PRs and branches
    - Render `BranchRow` for each branch in a group (after PRs)
    - Empty check considers both `pull_requests` and `branches`
    - Track branch items in `item_ids` as `f"branch:{branch.name}"`
13. Update `on_list_view_selected`: `BranchRow` → `webbrowser.open(branch.compare_url)`
14. Update `on_list_view_highlighted`: `BranchRow` → post `BranchSelected`

### Detail Pane & App

15. Add `show_branch(branch: CandidateBranch)` to `DetailPaneWidget` showing branch name, age, compare URL, and "*No PR yet — press Enter to create one.*" description; hide irrelevant sections (reviewers, labels, checks, timeline)
16. Add `@on(BranchSelected)` handler in `app.py` that calls `detail_pane.show_branch`

### Exports

17. Update `widgets/__init__.py` to export `BranchRow`, `BranchSelected`
18. Update `github/__init__.py` exports if needed

## Edge Cases

- **Default branch**: Dynamically excluded via `defaultBranchRef` in the GraphQL query (not hardcoded main/master)
- **Branch name encoding**: Branches with slashes (e.g., `feature/foo`) must be URL-encoded in compare URL
- **Fork workflows**: Branches pushed to forks won't appear when querying the upstream repo (known v1 limitation)
- **Large branch count**: Capped at 100 most-recent refs; 7-day recency filter narrows further
- **No repo configured**: Feature depends on existing required repo config — no special handling needed
