## Objective

Replace the hardcoded `version = "0.1.0"` in `pyproject.toml` with dynamic versioning derived from git tags using `hatch-vcs`. This eliminates manual version bumps â€” the version is determined at build/install time from the nearest git tag.

## Background

After T31 added `ghrd --version`, the version string comes from `importlib.metadata.version("gh-review-dashboard")`, which reads the value baked into package metadata at install time. Currently that value is the static `version` field in `pyproject.toml`. This means every release requires a manual edit to that field and a dedicated commit.

With `hatch-vcs`, the version is derived from `git describe --tags`:
- On tag `v0.2.0` â†’ version `0.2.0`
- 3 commits after `v0.2.0` â†’ version `0.2.1.dev3+g1a2b3c4`
- No tags â†’ version `0.0.0.dev1+gabcdef`

This gives dev builds self-describing versions and makes releases as simple as `git tag vX.Y.Z`.

## Requirements

1. Add `hatch-vcs` as a build dependency
2. Switch `pyproject.toml` from static `version` to dynamic versioning via VCS
3. `ghrd --version` continues to work, now showing the git-derived version
4. `get_version()` in `updater.py` continues to work (it already uses `importlib.metadata.version` with a `"dev"` fallback, which naturally handles editable installs without tags)
5. Tag the repo `v0.1.0` so existing state has a baseline version
6. Dev installs (`uv sync` / `pip install -e .`) produce descriptive dev versions
7. All existing tests pass without modification

## Plan

### 1. Modify `pyproject.toml`

Add `hatch-vcs` to build dependencies and configure dynamic versioning:

```toml
[build-system]
requires = ["hatchling", "hatch-vcs"]
build-backend = "hatchling.build"

[project]
name = "gh-review-dashboard"
dynamic = ["version"]           # replaces version = "0.1.0"
# ... rest unchanged

[tool.hatch.version]
source = "vcs"

[tool.hatch.build.hooks.vcs]
version-file = "src/gh_review_dashboard/_version.py"
```

The `version-file` hook auto-generates a `_version.py` file at build time containing `__version__ = "X.Y.Z"`. This is optional but useful for runtime access without `importlib.metadata`.

### 2. Add `src/gh_review_dashboard/_version.py` to `.gitignore`

This file is auto-generated at build time and should not be committed.

### 3. Verify `updater.py` needs no changes

`get_version()` already uses `importlib.metadata.version()` with a `PackageNotFoundError` fallback. This works with dynamic versioning â€” the metadata is populated at install time regardless of how the version was determined.

### 4. Create baseline tag

```bash
git tag v0.1.0
```

This gives `hatch-vcs` a starting point. Without any tags, the version would be `0.0.0.devN`.

### 5. Update tests

Existing `test_updater.py` and `test_main.py` version tests mock `importlib.metadata.version`, so they should pass without changes. Add a smoke test that `get_version()` returns a non-empty string without mocking (integration-level check that hatch-vcs is wired up).

### 6. Update README

Document the release workflow:
- Creating a release: `git tag vX.Y.Z && git push --tags`
- Note that dev builds show descriptive versions like `0.1.1.dev3+gabcdef`
