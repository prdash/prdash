## Objective

Surface `merge_state_status` as visible colored badges on PR rows and the detail pane, highlighting conflict, blocked, and behind states to help users identify PRs that need attention.

## Background

T38 added the `merge_state_status` field to the data model and used it for green-highlighting ready-to-merge PRs. However, other states like `DIRTY` (merge conflicts), `BLOCKED` (missing requirements), and `BEHIND` (needs rebase) are equally important for users to see at a glance. Displaying these as colored badges provides immediate visual feedback about PR health. Since the data is already fetched via the GraphQL query (from T38), no API changes are needed.

## Requirements

1. **Badge definitions**: Map `merge_state_status` values to badge labels and colors:
   - `DIRTY` -> "CONFLICT" badge in red
   - `BLOCKED` -> "BLOCKED" badge in yellow
   - `BEHIND` -> "BEHIND" badge in yellow
   - `UNSTABLE` -> "UNSTABLE" badge in yellow
   - `CLEAN` -> no badge (already handled by T38 green highlight)
   - `UNKNOWN` / other -> no badge
2. **PR row badges**: Add badge `Label` widget(s) to `PRRow.compose()` based on `pr.merge_state_status`
3. **Detail pane badges**: Show the badge in the detail pane metadata section alongside other PR metadata
4. **CSS styling**: Add CSS rules for each badge variant with appropriate background colors
5. **Tests**: Test badge rendering for each state, test no badge for CLEAN/UNKNOWN, test detail pane display
