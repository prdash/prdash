## Objective

PRs that match multiple query groups currently appear once in each matching group, cluttering the list. Implement a clean deduplication strategy so each PR appears in only one group, while still giving the user visibility into why a PR matched.

## Background

Query groups (e.g. "Direct reviewer", "Team reviewer", "Authored") run independent GitHub searches. A single PR can match multiple groups — for example, a PR you authored that also requests your review. Today the PR renders in every matching group, which is noisy and confusing.

## Requirements

1. **Primary-group assignment**: Each PR appears in exactly one query group. When a PR matches multiple groups, assign it to the highest-priority group (i.e. the first group in the configured list that matched it).
2. **Dedup by PR identifier**: Use the PR's unique identifier (node ID or `repo + number`) as the dedup key.
3. **Dedup happens at the display/data-assembly layer** — individual GitHub queries should remain unchanged so they each return complete, correct results independently.
4. **Cross-group badge (stretch)**: Optionally show a small indicator on a PR row (e.g. `[+2]`) when the PR also matched other groups, so the user knows it was multi-matched. This is nice-to-have.
5. **Preserve counts**: Group header counts should reflect the deduplicated PR count for that group.

## Design Notes

- The natural place to dedup is after all query group results are collected and before they are passed to `PRListWidget.update_groups()`.
- A simple algorithm: iterate groups in config order, track a `seen: set[str]` of PR identifiers, and filter each group's PR list to exclude already-seen PRs.
- Consider adding a helper function (e.g. `deduplicate_groups(results: list[QueryGroupResult]) -> list[QueryGroupResult]`) in `models.py` or a small utility module.
- Ensure refresh (both auto and manual) also deduplicates consistently.

## Acceptance Criteria

- [ ] A PR appearing in N query group results is shown in exactly one group in the UI
- [ ] The group it appears in is the first (highest-priority) matching group per config order
- [ ] Group header counts reflect post-dedup counts
- [ ] Underlying GitHub API queries are not altered
- [ ] Unit tests cover: no overlap, partial overlap, full overlap, empty groups, single group
- [ ] Manual QA: run the dashboard and confirm no duplicate PR rows
