## Objective

Create a Homebrew tap so users can install the dashboard via `brew tap ghrd/tap && brew install ghrd`.

## Background

Users on macOS expect `brew install` as a first-class install method. A personal Homebrew tap (separate GitHub repo) lets us distribute without meeting homebrew-core's acceptance bar. Releases are triggered by tagging a version in the main repo, which creates a GitHub Release with a source tarball that the formula points to.

### Prerequisites (may need separate work)

- **LICENSE file**: The project has no license yet. Homebrew formulas require a `license` field. A LICENSE file must exist before this task.
- **GitHub remote**: The repo has no GitHub remote configured. The main repo must be pushed to GitHub before creating releases or a tap.

## Requirements

### 1. GitHub Release workflow

- Add a GitHub Actions workflow (`.github/workflows/release.yml`) that triggers on version tags (`v*`)
- Workflow creates a GitHub Release from the tag with auto-generated release notes
- The release's source tarball (auto-created by GitHub) is what the formula downloads

### 2. Homebrew formula

- Create a formula file `Formula/ghrd.rb` (to be placed in the tap repo)
- Formula approach: Since this is a personal tap (not homebrew-core), use `pip install` into a virtualenv from the release tarball — avoids maintaining 20-30 resource blocks for transitive Python dependencies
- Formula structure:
  - `include Language::Python::Virtualenv`
  - `depends_on "python@3.12"`
  - `url` pointing to GitHub Release tarball
  - `sha256` of the tarball
  - `install` method: create virtualenv, pip install from tarball, symlink `ghrd` binary
  - `test` block: `assert_match version.to_s, shell_output("#{bin}/ghrd --version")` (depends on T31)
  - `license` field matching the project's LICENSE

### 3. Tap repo setup

- Create GitHub org `ghrd` (manual step — document in task)
- Create repo `ghrd/homebrew-tap` via `gh repo create`
- Push the formula to the tap repo
- Verify `brew tap ghrd/tap && brew install ghrd` works

### 4. Release helper script

- Add a `scripts/release.sh` (or document the process) for cutting a new release:
  1. Bump version in `pyproject.toml`
  2. Commit and tag (`git tag v0.1.0`)
  3. Push tag (`git push --tags`) — triggers the GitHub Actions release workflow
  4. After release is created: update formula SHA256 and version in the tap repo
- Consider: a GitHub Actions workflow in the tap repo that auto-updates the formula when a new release is published in the main repo (webhook/dispatch)

### 5. Documentation

- Update README with Homebrew install instructions
- Document the release process for maintainers

## Acceptance Criteria

- [ ] GitHub Actions release workflow triggers on `v*` tags and creates a Release
- [ ] Homebrew formula installs `ghrd` into a working state
- [ ] `brew tap ghrd/tap && brew install ghrd` works on a clean macOS machine
- [ ] `brew install ghrd` → `ghrd --version` prints the correct version
- [ ] `ghrd --update` works for Homebrew installs (or gracefully tells user to use `brew upgrade`)
- [ ] Release process is documented (version bump → tag → push → formula update)
- [ ] README includes Homebrew install instructions

## Notes

- **Org naming**: User will create a `ghrd` GitHub org. Tap repo: `ghrd/homebrew-tap`. Users type `brew tap ghrd/tap`.
- **Python version**: Formula depends on `python@3.12` (matches `requires-python = ">=3.12"` in pyproject.toml). May need updating if Python version requirements change.
- **T31 dependency**: The formula's `test` block uses `ghrd --version`, which is added in T31.
- **Future**: If the project is published to PyPI, the formula can switch to a simpler PyPI-based install with `poet`-generated resource blocks.
