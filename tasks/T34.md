## Objective

Replace the bulk `refs` GraphQL query with an Events API + GraphQL batch verification strategy so that "Ready to PR" branch detection works for repos with thousands of branches.

## Background

The original `BRANCH_REFS_QUERY` (from T30) fetched only the first 100 branches via GraphQL `refs` with `TAG_COMMIT_DATE` ordering. This ordering doesn't work correctly for branch refs (returns roughly reverse-alphabetical, not chronological). For large repos (e.g., 8,227 branches), the user's branches never appeared in the first 100 results, causing the "Ready to PR" group to show a header but no branches.

## Approach

Two-phase strategy: **Discover** then **Verify**.

1. **Discover**: REST `GET /users/{username}/events` returns the user's recent push/create events, identifying which branches they pushed to (1 API call per page, up to 3 pages, early stop if events > 7 days old).
2. **Verify**: A dynamically-built batched GraphQL query checks only those specific branches for open PRs, commit dates, and default branch status. Uses alias-based grouping (`r0`, `r1`, `b0_0`, `b0_1`, etc.) to handle multiple repos and branches in a single request.

This works for any repo size since we only query branches we know the user pushed to.

## Changes

### `src/gh_review_dashboard/github/queries.py`

- Removed `BRANCH_REFS_QUERY`
- Added `build_branch_verification_query(repo_branches)` — builds a dynamic GraphQL query with per-repo/per-branch aliases; sanitizes branch names (skips `"` and `\` to prevent injection); returns `(query_str, variables, alias_map)`

### `src/gh_review_dashboard/models.py`

- Removed `parse_refs_results`
- Added `parse_user_events(events, repos)` — parses REST events (PushEvent `refs/heads/...`, CreateEvent `ref_type=branch`) into `{repo_slug: {branch_names}}`; filters to configured repos; deduplicates
- Added `parse_branch_verification(data, alias_map)` — parses batched GraphQL response using alias_map; skips null refs (deleted), open PRs, old branches (>7 days), default branch; builds `CandidateBranch` objects with URL-encoded compare URLs

### `src/gh_review_dashboard/github/client.py`

- Added `_fetch_user_events(username)` — REST GET with pagination (up to 3 pages), 7-day early stop, error handling matching `execute_query` (401→AuthError, 403→GitHubAPIError, timeout→NetworkError)
- Rewrote `fetch_candidate_branches` — calls `_fetch_user_events` → `parse_user_events` → `build_branch_verification_query` → `execute_query` → `parse_branch_verification`
- Updated imports (removed `BRANCH_REFS_QUERY`, `parse_refs_results`; added new functions)

### `tests/test_models.py`

- Removed `TestParseRefsResults` (8 tests)
- Added `TestParseUserEvents` (7 tests): push/create event extraction, repo filtering, dedup, non-branch events ignored, empty events, empty repos allows all
- Added `TestParseBranchVerification` (7 tests): basic success, null ref, open PR, old branch, default branch, URL encoding, multiple repos

### `tests/test_github_client.py`

- Removed `_branch_refs_response` and `_make_ref_node` helpers
- Added `_make_push_event`, `_events_response`, `_branch_verification_response` helpers
- Rewrote `TestFetchCandidateBranches` (6 tests): single repo, multi repo, empty repos, no events, deleted branch, branch with open PR — mocking both REST events and GraphQL endpoints
- Rewrote `TestFetchAllGroupsWithBranches` (1 test): mixed PR + branch scenario with REST + GraphQL mocks
